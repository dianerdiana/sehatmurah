## Overview

This document captures the early brainstorming process used to shape SehatMurah into a healthcare booking platform. The goal was to identify the most important user problems, group the ideas into clear feature clusters, and prioritize the work that could realistically be completed within the project window.

## Step 1: Team Gathering and Problem Selection

### Stakeholder Pain Points

We discussed the needs of three core stakeholders:

- Patients want a fast way to find trusted doctors, compare availability, and book appointments without long queues.
- Doctors want a simple way to manage schedules, update profile details, and keep patient traffic organized.
- Administrators want a centralized system to verify doctors, manage users, and maintain platform quality.

### Selected Problem Statement

**How might we build a secure and user-friendly doctor appointment platform that helps patients find verified doctors, lets doctors manage their schedules efficiently, and gives administrators clear control over platform governance?**

## Step 2: Idea Listing and Grouping

The brainstormed ideas were grouped into four practical categories:

| Category | Ideas |
| -------- | ----- |
| Patient experience | Secure registration and login, doctor search, doctor detail pages, booking flow, appointment history, review submission, profile management. |
| Doctor workspace | Doctor profile management, availability and schedule updates, appointment monitoring, profile photo upload, clinic/location information. |
| Admin operations | Doctor approval, specialist management, user management, role-based access control, review moderation. |
| Platform foundation | JWT authentication, password hashing, server-side validation, MongoDB persistence, media upload support, and static file serving. |

## Step 3: Idea Prioritization

The ideas were then prioritized based on value, feasibility, and the available project timeline.

| Feature Initiative | Priority | Reason |
| ----- | ----- | ----- |
| Authentication and role-based access | Core MVP | Required to secure patient, doctor, and admin access. |
| Doctor search and filtering | Core MVP | Solves the main discovery problem for patients. |
| Appointment booking flow | Core MVP | Defines the main transaction of the application. |
| Doctor profile and schedule management | Core MVP | Allows doctors to maintain availability and practice data. |
| Patient and doctor profile management | Core MVP | Supports the basic account lifecycle for all users. |
| Specialist management | Core MVP | Helps organize doctor discovery and categorization. |
| Review management | Core MVP | Builds trust and supports post-appointment feedback. |
| User administration and doctor approval | Core MVP | Keeps the platform controlled and reliable. |
| Media upload support | Supporting feature | Useful for profile photos and platform assets already supported in the stack. |
| Full medical document vault | Future roadmap | Valuable, but better expanded after the core booking flow is stable. |
| In-app notifications | Future roadmap | Helpful for reminders and status updates, but not required for the MVP. |
| Telemedicine | Future roadmap | High value, but too large for the current capstone window. |
| Payment integration | Future roadmap | Can be added after the appointment flow is proven. |
| AI symptom checker | Future roadmap | Interesting enhancement, but not part of the first release. |

## Summary

The brainstorming phase established a clear direction: build the booking platform first, then extend it with richer healthcare services. This prioritization keeps the project realistic while still leaving room for future expansion.
